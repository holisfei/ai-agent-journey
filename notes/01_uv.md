# pip/poetry/conda/uv

### 1991年 pip → 只装包，不管环境/版本/锁定

最原始的Python依赖管理，只管安装，安装在了系统，所有项目公共一份
- 没有Python版本隔离管理（需要单独配合pyenv）
- 没有独立的虚拟环境管理（需要单独配合venv/virtualenv）
- 没有依赖锁定
- 不解决依赖冲突

### 2007年 Python虚拟环境（virtual environment）→ 解决「项目隔离」

Python的项目和iOS的项目不一样，iOS每个项目都有自己的独立依赖库(pods, SPM)，但是Python的依赖库都会安装到系统里(pip install xxx),默认是在全局装包

pip 会把包装到【当前激活的python解释器】的【site-packages】下，如果没有创建虚拟环境，那当前的python就是系统python

如果每个项目需要依赖不同版本的包，那这时候就会冲突。

【虚拟环境】本质就是临时切换【当前激活的python解释器】
- 把【python解释器】从系统指向到项目.venv/bin/python

所以Python项目需要自己独立的包依赖管理和Python版本管理
- 独立的Python运行环境
- 独立的Python依赖管理

### 2008年  pyenv → 解决「Python 版本管理」

### 2012年 conda → 数据科学圈另起炉灶

跨多语言的环境管理工具，不止可以装Python，还可以装R、C++ 库、CUDA、甚至非 Python 的二进制（如 ffmpeg）
- 在 GPU / 机器学习环境里几乎不可替代

### 2018年 poetry → 第一次尝试一体化，但慢且不管 Python 版本

依赖管理 + 虚拟环境 + 打包发布
- 引入了 pyproject.toml 作为项目的标准配置文件
- 自动解决依赖冲突
- 锁文件机制完善

缺点
- 不能下载 Python 版本，只能使用现有的Python版本（poetry env use python3.13）
- 是用用Python写的，速度慢

### 2024年 uv → Rust 重写，真正的一体化 + 极速

pip(安装依赖) + poetry(兼容poetry) + pyenv(Python环境版本) + venv(虚拟环境) 的全功能替代品。

官网：https://uv.oaix.tech/getting-started/features/#_3

uv做的事情
- uv init / uv add，uv会在根目录创建 .venv目录
    - bin：可执行文件，核心是python这个解释器本身（其实是软链接或拷贝到项目目录的 Python 副本），还有 pip、以及你装的库提供的其他命令行工具（比如装了 pytest 后会有 bin/pytest）
    - lib：你装的所有依赖库的 Python 代码真正存放的地方。
- uv run 就是使用了上面独立的环境和依赖
    - uv run xxx.py 等价于 .venv/bin/python xxx.py，再加上「跑之前自动 uv sync 保证环境最新」。所以 uv 项目里你几乎不需要手动 activate 虚拟环境——这是 uv 比 poetry 体验好的地方之一。
    - 当你在终端跑 python xxx.py 而不是 uv run xxx.py 时，用的是系统 Python，

把 bin 理解为「这个项目专属的命令」，lib 理解为「这个项目专属的库代码」，更贴近实际。

pyproject.toml 和 uv.lock
- pyproject.toml：保存了项目概览：Python运行环境，所有依赖库
- uv.lock：对当前Python运行环境，所有依赖库做了指定版本的锁定记录

开发依赖和运行依赖：Python的依赖库分为开发依赖和运行依赖
- 开发依赖：只在开发的时候依赖的库，uv add --dev xxx
- 运行依赖：上线时依赖的库，uv add xxx

uv常用命令

```
uv python install xxx        # 安装指定python
uv python list               # 查看可以安装的python
uv python find               # 查看本地已经安装的python

uv init                      # 用uv初始化一个python项目
uv add xxx                   # 安装python依赖库
uv sync                      # 同步项目依赖环境
uv run xxx.py                # 用uv运行一个py文件

uv remove xxx                # 卸载依赖
uv add --dev pytest          # 加开发依赖（不会装到生产）
uv add 'httpx>=0.27'         # 指定版本约束
uv tree                      # 看依赖树（哪个库依赖哪个）
uv lock                      # 只更新锁文件，不安装
uv lock --upgrade            # 升级所有依赖到允许范围内的最新版
uv run --python 3.11 xxx.py  # 临时用某个 Python 版本运行
uv pip install xxx           # 用 pip 兼容模式装包（救急用）
```

python指定环境：
```bash
uv python install 3.13 # 装 uv 自己管理的独立 Python
echo "3.12" > .python-version # 锁定项目 Python 版本
uv venv --python 3.13 # 显式用这个版本建虚拟环境
```

### uv 和 conda python环境冲突问题

```uv venv --python 3.13```默认会用系统能找到的第一个 Python 来创建虚拟环境。如果当时 conda 激活着, 那么 uv 创建的 .venv 是基于 conda 的 Python 的，表面看路径在 .venv/bin/, 但底层链接到 conda。

问题定位：
```bash
# 1. 看 .venv 里到底有没有某个依赖(比如 selectolax)
ls .venv/lib/python*/site-packages/ | grep -i selectolax

# 2. 看 .venv 里的 pytest 到底用哪个 Python 跑
cat .venv/bin/pytest | head -2

# 3. 看 .venv 里 python 真实指向哪
ls -la .venv/bin/python*
# .venv/bin/python -> /opt/miniconda3/bin/python3  # 这里的指向就是错误的，指向了miniconda3
# .venv/bin/python3 -> python
# .venv/bin/python3.13 -> python
```

修复：

```bash
cd 根目录

# 1. 暂时关闭 conda 自动激活
conda config --set auto_activate_base false 2>/dev/null
conda deactivate 2>/dev/null

# 2. 删掉受污染的 .venv
rm -rf .venv

# 3. 让 uv 装一个干净的 Python 3.12(完全独立于 conda)
uv python install 3.12

# 4. 用这个干净的 Python 3.12 重建 .venv
uv venv --python 3.12

# 5. 安装所有依赖
uv sync

# 6. 验证 python 指向(关键!)
echo "--- python 指向 ---"
ls -la .venv/bin/python

# 7. 验证依赖装上了
echo "--- 验证依赖 ---"
uv run python -c "import selectolax, httpx, pydantic, typer, rich; print('all good')"

# 8. 跑测试
uv run pytest
```
